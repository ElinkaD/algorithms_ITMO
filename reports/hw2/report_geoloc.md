# Отчет по геопоиску Lat/Lng

## Задание

Нужно было реализовать систему геопоиска по координатам `Lat/Lng`, которая:

- хранит объекты вида `ID`, `Lat`, `Lng`;
- поддерживает точный поиск по geohash-ячейке;
- поддерживает поиск ближайших объектов в заданном радиусе;
- использует индекс на основе `B-tree` и `geohash`;
- имеет baseline в виде полного перебора `FullScan`.

## Что реализовано

Реализация сделана на Go и полностью работает в оперативной памяти.

Основные части решения:

- `EncodeGeohash(lat, lng, precision)` для кодирования координат;
- `B-tree` для хранения уникальных geohash-ключей;
- `map[string][]GeoObject` для хранения bucket-ов с объектами;
- `Insert`, `SearchExact`, `SearchNearby`;
- baseline `FullScan`;
- генератор случайных данных;
- функциональные тесты и бенчмарки.

Полезные файлы:

- код индекса: [../../hw2/index.go](../../hw2/index.go)
- geohash и геометрия окна поиска: [../../hw2/geohash.go](../../hw2/geohash.go)
- B-tree: [../../hw2/btree.go](../../hw2/btree.go)
- тесты: [../../hw2/index_test.go](../../hw2/index_test.go)
- бенчмарки: [../../hw2/benchmark_test.go](../../hw2/benchmark_test.go)

## Принятые предположения

- `SearchExact` трактуется как поиск bucket-а по geohash точки, а не как поиск объектов с побитово равными координатами.
- Для расстояния используется формула haversine.
- Точность geohash задается параметром `precision`.
- Итоговое попадание объекта в ответ nearby-поиска всегда проверяется по реальному расстоянию.

## План работы

Работа шла по таким шагам:

1. Сначала был собран минимальный каркас: модель объекта, geohash, B-tree и buckets.
2. Затем были реализованы `Insert`, `SearchExact` и первая версия `SearchNearby`.

4. Дальше были добавлены генератор данных и тесты на случайных наборах.

6. Затем были добавлены бенчмарки, сохранение артефактов, графики и профилирование.

## Трудности и как они решались

Главная сложность была в `SearchNearby`.

Первая версия nearby-поиска строилась через центральную geohash-ячейку и соседние ячейки. На случайных тестах это дало расхождения с `FullScan`: часть точек терялась на границах ячеек, около полюсов и около границы `-180/180`.

В ходе отладки стало понятно:

- если радиус поиска больше размера одной geohash-ячейки, простой обход соседей уже не всегда надежен;
- около полюсов меняется охват по долготе;
- около международной линии перемены дат интервал по долготе может разрываться.

Из-за этого работа шла в два этапа:

1. Сначала была сделана максимально надежная версия nearby-поиска через пересечение окна поиска со всеми занятыми geohash-ячейками. Этот вариант был удобен, чтобы добиться корректности и пройти тесты.
2. После первых бенчмарков стало видно, что такая версия на больших данных проигрывает `FullScan`, потому что перебирает слишком много buckets. После этого nearby-поиск был переписан: теперь он строит только набор соседних geohash-ячеек, покрывающих окно поиска, и работает только с ними.

Именно после этого nearby-поиск стал уже не только корректным, но и заметно быстрее baseline.

## Генерация данных

Данные синтетические.
Случайные точки генерируются программно в диапазонах:

- `lat in [-90, 90]`
- `lng in [-180, 180]`

Размеры наборов данных для бенчмарков:

- `1000`
- `10000`
- `50000`
- `100000`

Такой вариант удобен, потому что:

- легко воспроизводится;
- хорошо подходит для нагрузочных тестов;
- позволяет контролировать размер набора и параметры запросов.

## Основные результаты

Характерные числа после оптимизации nearby-поиска:

- `SearchExact size=100000` -> около `253 ns/op`
- `SearchNearby size=100000 precision=4 radius=1000` -> около `453 ns/op`
- `SearchNearby size=100000 precision=4 radius=50000` -> около `3822 ns/op`
- `SearchNearby size=100000 precision=5 radius=50000` -> около `94868 ns/op`
- `FullScan size=100000 radius=1000` -> около `7.09 ms/op`
- `FullScan size=100000 radius=50000` -> около `7.17 ms/op`

Главные наблюдения:

- `SearchExact` очень быстрый и почти не зависит от размера набора данных.
- Вставка растет вместе с числом объектов, что ожидаемо для in-memory индекса.
- После оптимизации `SearchNearby` стал заметно быстрее `FullScan`.
- При `precision=4` nearby-поиск обычно быстрее и дешевле по памяти, чем при `precision=5`.
- При `precision=5` и большом радиусе запроса nearby-поиск становится тяжелее, потому что приходится обходить больше geohash-ячеек.

## Вывод

В этой работе был реализован геоиндекс на основе `geohash + B-tree + buckets`.

Самый важный практический результат такой:

- сначала была получена корректная реализация nearby-поиска;
- затем она была оптимизирована до обхода только соседних geohash-ячеек;
- после этого nearby-поиск стал заметно быстрее линейного baseline `FullScan`.

При этом по результатам измерений видно, что выбор `precision` влияет на поведение системы:

- меньшая точность дает более дешевый nearby-поиск;
- большая точность дает более детальное разбиение, но на широких запросах увеличивает число ячеек и расходы по памяти.

То есть итоговое решение не только работает корректно, но и показывает ожидаемое поведение по времени и памяти на разных режимах поиска.
